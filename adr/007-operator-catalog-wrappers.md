# ADR-007: Operator Catalog Wrappers via Timoni-Generated Schemas

## Status

Accepted

## Context

The catalog already ships three ecosystem integrations that wrap upstream Kubernetes operator CRDs: `cert_manager/`, `k8up/`, and `ch_vmm/`. Adding ClickHouse ClickStack observability support requires catalog coverage for three more operators — MongoDB Community, Altinity ClickHouse, and OpenTelemetry Collector — so that application modules can compose typed custom resources without reinventing schemas.

Each operator exposes a substantial CRD surface. `ClickHouseInstallation.spec.configuration` alone spans hundreds of nested fields; `OpenTelemetryCollector.spec.config` is a free-form collector pipeline definition; `MongoDBCommunity.spec` covers replica set, TLS, Prometheus, and user management. Two patterns exist in the repo today:

1. **Structured transformation** (`cert_manager/`) — a curated OPM-specific schema maps to the upstream CRD field-by-field inside the transformer. Good ergonomics, but every upstream field addition requires manual work.
2. **Open passthrough** (`k8up/`) — schemas carry `...` on every struct and transformers copy `spec` verbatim. Fast to implement, but downstream consumers lose CUE field-level validation.
3. **Timoni-generated schemas** (`ch_vmm/`) — `timoni mod vendor crd` ingests upstream CRD YAML and emits typed CUE under `schemas/…/types_gen.cue`. Transformers remain passthrough; validation comes from the generated types.

The three new operators have deep, actively evolving CRD surfaces where hand-modeling is impractical, but validation matters because ClickStack mis-configuration is hard to debug in running clusters.

## Decision

Wrap the MongoDB Community, Altinity ClickHouse, and OpenTelemetry operator CRDs using the `ch_vmm/` pattern: vendored CUE types generated by `timoni mod vendor crd`, referenced by passthrough transformers. Each new catalog module (`mongodb_operator/`, `clickhouse_operator/`, `otel_collector/`) follows the same directory layout:

```text
<operator>/v1alpha1/
  cue.mod/module.cue
  resources/<category>/<kind>.cue               # OPM #Resource + #Component wrappers
  providers/kubernetes/provider.cue             # Provider registry
  providers/kubernetes/transformers/*.cue       # Passthrough transformers + tests
  schemas/<operator>/<api-group>/<version>/types_gen.cue  # Timoni output, do not edit
  README.md                                     # Upstream source, regeneration pipeline
  INDEX.md                                      # Generated
```

The README of each module documents the exact `curl` → `yq`/`csplit` → `timoni mod vendor crd` pipeline used to (re)generate schemas, the upstream source URL, and the pinned release tag. Schemas are regenerated when upstream CRDs change; they are never edited by hand.

Operator Deployments, ServiceAccounts, ClusterRoles, and CRD install manifests live in `modules/<operator>/` — not in the catalog. The catalog only publishes typed resource definitions.

**Alternatives considered:**

- **Structured transformation per operator.** Matches `cert_manager/`'s elegance but requires deep field-by-field mappings across three large operators. Rejected: implementation cost is weeks, and upstream drift would silently break consumers.
- **Pure passthrough (k8up style).** Faster than timoni, but downstream modules cannot catch common typos (`spec.memebers` vs `spec.members`) until runtime. Rejected for ClickStack specifically because misconfigured ClickHouse/MongoDB state corrupts persisted data.
- **One combined `clickstack_operators/` catalog.** Mirrors upstream's `clickstack-operators` Helm chart bundling MongoDB + ClickHouse operators. Rejected: operator catalogs should stay single-purpose so non-ClickStack users can consume one operator without pulling in the others.

## Consequences

**Positive:** Downstream modules get full CUE field-level validation for every CR they emit. Schema drift across operator releases is a mechanical regeneration (`timoni mod vendor crd -f`) rather than a manual diff. The three new catalogs remain independently reusable — a future non-ClickStack module can consume `otel_collector/` alone without pulling MongoDB or ClickHouse.

**Negative:** Timoni becomes a hard dependency for contributors who need to refresh CRDs. The vendored `types_gen.cue` files are large (hundreds of KB for deep schemas) and inflate the catalog's on-disk footprint. CUE-level validation is only as strict as the upstream CRD's OpenAPI schema — operators that leave fields as `x-kubernetes-preserve-unknown-fields: true` effectively fall back to passthrough anyway.

**Trade-off:** Regenerating schemas requires running an external command pipeline outside the normal `task` workflow. This is acceptable because upstream CRD bumps are infrequent and the pipeline is documented in each module's README.
